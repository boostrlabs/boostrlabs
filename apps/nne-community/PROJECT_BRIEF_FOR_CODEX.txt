==================================================
1. PRODUCT VISION
==================================================

NNE Community is a gamified platform for emerging artists and NNE fans.

Users complete quests that create real value for the NNE ecosystem:
- Interact with NNE artists on social platforms.
- Participate in listening quests.
- Answer song trivia.
- Refer real artists.
- Attend lives or events.
- Create content.
- Complete future community missions.

The user receives NNE Credits after completing or passing a quest.

NNE Credits are NOT cryptocurrency and should not be presented as money.
They represent verified participation and contribution inside the NNE ecosystem.

Users can exchange NNE Credits for real NNE services:
- Song feedback.
- Master.
- Vocal mix.
- Beat lease.
- Cover art.
- Exclusive beat.
- Full production in later versions.

The app must feel like a premium dark videogame/community operating system,
not like a survey website or basic loyalty program.

3. IMPORTANT PRODUCT RULES
==================================================

1. Credits should only be added after one of these events:
   - Admin approves submitted evidence.
   - User passes an automatically graded quest.
   - Backend confirms a referral or another event.

2. Screenshot quests should become "pending", not completed, after submission.

3. Listening quests should not rely on screenshots.
   They should use:
   - Official song link.
   - Minimum waiting/listening timer.
   - Random questions from a larger question bank.
   - Required passing percentage.
   - Limited retries and cooldown in the backend later.

4. The frontend must NEVER reveal correct trivia answers before submission.

5. Correct answers must eventually come from the backend.
   Do not ship correctOptionIndex to the browser in production.
   It is included in mockData only to demonstrate the interaction.

6. NNE Credits must use an immutable transaction ledger in the backend.
   Never update only a numeric balance without recording why it changed.

7. Reward redemption should create a redemption request.
   Do not instantly mark a service as delivered.

8. User progress should remain visible and aspirational:
   - Locked rewards.
   - Next level.
   - XP progress.
   - Streak.
   - Achievements later.

6. LISTENING QUEST PRODUCTION FLOW
==================================================

Recommended secure flow:

1. Frontend calls:
   POST /api/quests/:id/start

2. Backend creates quest attempt and returns:
   - attempt_id
   - song_url
   - unlock_at
   - trivia_session_available_at

3. User opens song link.

4. Frontend waits until unlock_at.
   This timer is only UX. Backend must also enforce time.

5. Frontend calls:
   POST /api/quests/:id/trivia-session

6. Backend randomly selects questions and options.
   Backend does NOT return correct answers.

7. User submits answers:
   POST /api/trivia-sessions/:id/submit

8. Backend grades answers.

9. If passed:
   - Mark completion completed.
   - Add credit transaction.
   - Update balance.
   - Create feed event if public.
   - Return result and new balance.

10. If failed:
   - Increase attempt count.
   - Apply cooldown.
   - Return next_retry_at.
   - Never return the full correct answer set.

10. DEFINITION OF MVP READY FOR USERS
==================================================

The MVP is ready for private testing when:

- Users can register and log in.
- Admin can create quests.
- Users can start quests.
- Users can upload screenshots.
- Admin can approve/reject evidence.
- Approved quests generate credit ledger entries.
- Listening quests generate secure random trivia.
- Backend grades trivia.

- Admin can manage reward requests.
