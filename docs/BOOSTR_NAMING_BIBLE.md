# BOOSTR Labs Naming Bible

Status: ACTIVE PRODUCT LANGUAGE
Last updated: 2026-07-08

## Core rule

Use names people understand.

BOOSTR can sound proprietary, but not confusing. If a name does not immediately explain the surface, use the clearer name.

## Product clarity rule

If the user or client cannot understand a name in 2 seconds, simplify it.

Examples:

- Use `Leads`, not `Signal Inbox`.
- Use `Dashboard`, not `Workspace Core`.
- Use `Partners`, not `Partner Grid`.
- Use `BOOSTR CORE`, not `Mother OS`.
- Use `BOOSTR Audit`, not `BOOSTR Intake`.

## Branded route rule

Branded routes are not generic BOOSTR modules.

- JANKO/WESTDETRO should feel like JANKO first.
- 82NGEL should feel like 82NGEL first.
- BOOSTR appears as powered-by infrastructure, not the visual owner.

## Forbidden UI words

Avoid visible internal language:

- backend pending
- Codex
- Antigravity
- mockup
- invented demo data
- ready for leads
- preview systems
- user login shell
- temporary access token outside internal testing
- TODO
- placeholder

## Copy pattern

Preferred card structure:

```text
Leads
24
Review
```

or:

```text
Dashboard
Active
Open
```

Avoid long explanations in UI.

## EN/ES rule

Product names stay fixed when they are brand/product names.

Translate the surrounding UI only:

- buttons
- helper labels
- validation
- status
- form fields
- table headers

Example:

English:

- Open Dashboard

Spanish:

- Abrir Dashboard

## Final naming test

A name passes if it is:

- clear
- short
- premium
- easy to explain
- not generic in the wrong place

A name fails if it sounds cool but no one understands what it does.
